Wire --permission-mode into main.py's CLI, reusing permissions.run_agent_with_mode

Exposes the fully-tested permissions.py from the previous commit at the
CLI entry point: python main.py --permission-mode {plan,accept_edits,
auto,dont_ask,bypass,default}.

Backward-compatibility guarantee, deliberately verified not just assumed:
omitting --permission-mode entirely takes the EXACT prior code path (a
plain run_agent(user_input, verbose=True, confirm=confirm) call, never
touching permissions.py) rather than always routing through
run_agent_with_mode(mode=DEFAULT) -- because PermissionEngine's
system_prompt_suffix() appends a note to the system prompt for every
mode including DEFAULT, and routing unconditionally would have subtly
changed the prompt sent to the model even with unchanged confirm()
behavior. _parse_permission_mode returns Python None (not
PermissionMode.DEFAULT) specifically to distinguish "flag not given"
from "flag given, value is default" -- main() branches on that
distinction.

A real bug caught by the test suite, not by inspection: the first
draft's sys.exit(1) error paths in _parse_permission_mode had no return
statement after them -- harmless in real use (sys.exit raises
SystemExit), but a test that mocks sys.exit to make it inspectable
exposed that execution would otherwise fall through to argv[idx+1] and
crash with IndexError instead of exiting cleanly. Fixed with an
explicit return after each sys.exit(1) call.

Also added explicit stderr warnings (not silent behavior) for --dry-run
combined with bypass/accept_edits/auto modes, since those three modes
partially or fully bypass --dry-run's decline-and-report confirm
callable by design (see permissions.py) -- a user combining these
flags is very likely assuming --dry-run's protection is unconditional
across every mode, which is the exact silent-surprise class of bug
this project has been careful to avoid elsewhere (e.g. the
SYSTEM_PROMPT confirmation-bypass bug found and fixed earlier).

Verified: test/main_permission_mode_cli_test.py (9 scenarios, no LLM,
mocked run_agent/run_agent_with_mode to prove main() takes the correct
call path in both the flag-given and no-flag cases) plus real
subprocess runs of the actual main.py CLI (startup+exit for a valid
mode and for no flags, and exit-code-1 error paths for a missing value
and an unknown mode value). All 12 pre-existing test files still pass
unchanged.
